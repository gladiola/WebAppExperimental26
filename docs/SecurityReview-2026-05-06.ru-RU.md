## Подробные обнаружения

### ✅ Подтверждённые исправления от 2026-05-05

#### 1. Повторное использование IV AES-GCM — Исправлено

**Файл:** `Models/Main_Objects/Nonce.cs`

Генерация nonce на основе AES-GCM была полностью заменена. `Nonce.GenerateSecureNonce()` теперь вызывает `RandomNumberGenerator.Fill(randomBytes)` для 16 случайных байт и возвращает строку Base64. Нет зависимости от Key Vault, нет IV, нет шифрования — именно правильный подход для CSP-nonce.

---

#### 2. Значения Nonce больше не записываются в журнал — Исправлено

**Файлы:** `Services/NonceMiddleware.cs`, `Services/NonceRefresherService.cs`

Оба файла теперь записывают только сообщения о статусе (`"Nonce retrieved for request."`, `"Nonce generated successfully."`) и никогда само значение nonce.

---

#### 3. Удалены жёстко закодированные резервные Nonce — Исправлено

**Файл:** `Services/OptimizedNonceMiddleware.cs`

Все три жёстко закодированные литеральные строки (`"bootstrap-nonce-placeholder"`, `"fallback-nonce"`, `"error-fallback-nonce"`) заменены вызовами `Nonce.GenerateSecureNonce()` как в обычном, так и в резервном пути обработки исключений.

---

#### 4. Потокобезопасный словарь Nonce — Исправлено

**Файл:** `Services/NonceCatalogService.cs`

`Dictionary<string, Nonce>` заменён на `ConcurrentDictionary<string, Nonce>`. `GetANonce` теперь использует единственный атомарный вызов `TryGetValue` вместо двухэтапной проверки и последующего поиска.

---

#### 5. Проверка издателя mTLS теперь функциональна — Исправлено

**Файл:** `Extensions/ServiceCollectionExtensions.cs`, `Models/Settings/MtlsSettings.cs`

Закомментированный блок проверки издателя заменён вызовом `mtlsSettings.IsIssuerAllowed(issuer)`, который выполняет поиск подстроки без учёта регистра в `AllowedIssuers`. Когда список пуст (не настроен), метод возвращает `false`, отклоняя все сертификаты (fail-closed).

---

#### 6. Проверка отзыва mTLS включена по умолчанию — Исправлено

**Файл:** `Models/Settings/MtlsSettings.cs`

`CheckCertificateRevocation` теперь по умолчанию равен `true`. В `appsettings.template.json` также указано `"CheckCertificateRevocation": true`.

---

#### 7. Заглушка OCSP теперь не проходит в закрытом состоянии — Исправлено

**Файл:** `Services/OcspValidationService.cs`

`PerformOcspValidationAsync` теперь возвращает `IsValid = false` с `OcspStatus.Error` и записывает ошибку, вместо того чтобы молча возвращать `IsValid = true`. Включение OCSP в конфигурации теперь будет отклонять все сертификаты до тех пор, пока не будет предоставлена реальная реализация.

---

#### 8. Аутентификация и авторизация включены по умолчанию — Исправлено

**Файл:** `Models/Settings/FeatureFlags.cs`

`EnableAzureAd` и `EnableAuthorization` теперь оба по умолчанию равны `true` в классе `FeatureFlags`. `appsettings.json` также устанавливает оба значения в `true`.

---

#### 9. Заголовки безопасности применяются до маршрутизации — Исправлено

**Файл:** `Program.cs`

`UseNonceAndSecurityHeadersAsync` и `UseStandardSecurityHeaders` теперь вызываются до `UseRouting`, `UseAuthentication` и `UseAuthorization`. Все ответы, включая короткие замыкания 401/403, получают заголовки безопасности.

---

#### 10–15. Cookie, Content-Type, AllowedHosts, Nonce в макете, Referrer-Policy — Исправлено

**Файлы:** `Extensions/ServiceCollectionExtensions.cs`, `Extensions/ApplicationBuilderExtensions.cs`, `Views/Shared/_Layout.cshtml`, `appsettings.json`

- Куки сессии теперь устанавливает `CookieSecurePolicy.Always` и `SameSiteMode.Strict`.
- Некорректный безымянный заголовок `Set-Cookie` удалён.
- Глобальное переопределение `Content-Type: text/html` удалено.
- `AllowedHosts` в `appsettings.json` теперь `"localhost;127.0.0.1"`; шаблон использует `"{{YOUR_HOSTNAME}}"`.
- Все три тега `<script>` в `_Layout.cshtml` теперь включают `nonce="@Context.Items["Nonce"]"`.
- `Referrer-Policy: strict-origin-when-cross-origin` теперь добавляется `UseStandardSecurityHeaders`.

---

#### 16–19. Журналирование PII, журнал строки подключения, заглушки Key Vault, X-XSS-Protection — Исправлено

**Файлы:** `Services/LoggingHelper.cs`, `Extensions/ServiceCollectionExtensions.cs`, `Extensions/ApplicationBuilderExtensions.cs`

- Все PII (OID, электронная почта, имя, SID, роли) теперь хешируются с помощью HMAC-SHA256 через `LoggingHelper.HashPii()` перед записью в журналы. Стабильный ключ HMAC может быть предоставлен через `Logging:PiiHmacKey` в конфигурации; случайный ключ для каждого процесса используется, если не настроен.
- Оператор журнала Cosmos DB теперь только подтверждает наличие строки подключения (`!string.IsNullOrEmpty`), а не её содержимое.
- `AzureKeyVaultCertificateOperations` теперь выбрасывает `InvalidOperationException` при запуске, когда сертификат равен null, вместо того чтобы молча возвращать фиктивные значения.
- `X-XSS-Protection` теперь установлен в `"0"` (отключая устаревший XSS-аудитор), в соответствии с современными рекомендациями браузеров.

## 🟠 Высокая

### 20. NonceRefresherService сохраняет неиспользуемые зависимости конструктора Key Vault

**Файл:** `Services/NonceRefresherService.cs`

`NonceRefresherService` по-прежнему объявляет параметры конструктора для `IKeyVaultSettingsService`, `INonceEncryptionSettingsService`, `IAzureADSettingsService` и `IAzureKeyVaultOperationsService`. Поскольку генерация nonce была упрощена для прямого использования `RandomNumberGenerator`, ни одна из этих зависимостей не используется.

**Риск:** Когда `EnableNonceServices = true` и `EnableKeyVault = false` (значение по умолчанию), эти сервисы не регистрируются в контейнере DI, что вызывает `InvalidOperationException` во время выполнения при первом разрешении сервиса nonce. Это фактически условие отказа в обслуживании, вызванное конфигурацией по умолчанию. Класс `FeatureFlags` по умолчанию устанавливает `EnableNonceServices = true`, поэтому любая среда, опирающаяся исключительно на значения по умолчанию класса (без переопределений `appsettings.json`), не сможет запуститься.

**Рекомендация:** Удалить четыре неиспользуемых параметра конструктора и соответствующие приватные поля из `NonceRefresherService`. Сервису требуются только `ILogger<NonceRefresherService>`, `ILoggerFactory` и `INonceCatalogService`.

## 🟡 Средняя

### 21. Внутренний кэш OcspValidationService использует небезопасный для потоков Dictionary

**Файл:** `Services/OcspValidationService.cs` (строка 47)

```csharp
private readonly Dictionary<string, CachedOcspResponse> _cache;
```

`Dictionary<TKey, TValue>` не является потокобезопасным для параллельных чтений и записей. Если `OcspValidationService` зарегистрирован как синглтон (или если один экземпляр совместно используется между запросами любым другим механизмом), параллельные проверки OCSP могут повредить кэш, вызывая потерю записей, выброс исключений или возврат устаревших данных.

**Рекомендация:** Заменить `Dictionary<string, CachedOcspResponse>` на `ConcurrentDictionary<string, CachedOcspResponse>`. Обновить вызов `_cache.Remove` (строка 103) на `_cache.TryRemove`.

## 🔵 Низкая / Информационная

### 22. Заглушка проверки OCSP — не проходит в закрытом состоянии, но не реализована

**Файл:** `Services/OcspValidationService.cs` (строки 157–173)

`PerformOcspValidationAsync` по-прежнему является заглушкой. Исправление в обнаружении #7 правильно изменило поведение с «всегда действительный» на «всегда недействительный (fail-closed)». Однако метод по-прежнему не является реальной реализацией OCSP. Пока `EnableOcspValidation = false` (по умолчанию), это не оказывает влияния на производственную среду. Перед включением OCSP в любой среде должен быть реализован OCSP-клиент производственного качества.

---

### 23. mTLS с пустым AllowedIssuers отклоняет все клиентские сертификаты

**Файл:** `Models/Settings/MtlsSettings.cs`

Когда `ValidateClientCertificateIssuer = true` (по умолчанию) и `AllowedIssuers` пуст (также по умолчанию при отсутствии конфигурации), `IsIssuerAllowed()` возвращает `false`, что приводит к отклонению всех клиентских сертификатов. Это правильное поведение fail-closed, но оно не задокументировано достаточно заметно. Операторы, включающие mTLS без тщательного изучения шаблона, могут обнаружить, что все клиентские подключения отклоняются без очевидного объяснения.

**Рекомендация:** Добавить предупреждающее сообщение в журнал при запуске, когда `ValidateClientCertificateIssuer = true` и `AllowedIssuers` пуст.

---

### 24. OcspSettings.ServerUnavailableBehavior по умолчанию «Warn»

**Файл:** `appsettings.template.json` (строка 134), `Services/OcspValidationService.cs`

Параметр `ServerUnavailableBehavior` по умолчанию установлен в `"Warn"` в шаблоне, что позволяет запросам проходить, когда сервер OCSP недоступен. Для сред с высокими требованиями к безопасности это должно быть `"Fail"`, чтобы сбои сервера OCSP не снижали незаметно проверку отзыва сертификатов.

**Рекомендация:** Чётко задокументировать три варианта (`Fail`, `Allow`, `Warn`) в шаблоне и рассмотреть возможность изменения значения по умолчанию на `"Fail"` в соответствии с принципом минимальных привилегий.

## Общая оценка

Приложение устранило все критические и высокосерьёзные уязвимости из предыдущей проверки. Текущие обнаружения ограничиваются одной высокосерьёзной проблемой конфигурации/DI (обнаружение #20) и информационными элементами более низкой серьёзности. Уровень безопасности существенно улучшился. Рекомендуется немедленное действие для обнаружения #20 (неиспользуемые зависимости DI в NonceRefresherService), так как это может помешать запуску приложения при конфигурации по умолчанию.
